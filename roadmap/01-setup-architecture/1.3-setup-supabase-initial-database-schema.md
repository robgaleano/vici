## Objective

Set up Supabase as VICI's initial backend, defining a minimal but solid schema for auth, player profile, teams, activities, zones, and wallet so the base can grow without prematurely mixing economy, ranking, and territorial logic.

## Decisions Made So Far

1. **Recommended identity pattern**: `auth.users` will be the canonical identity, and `public.profiles` will be the 1:1 product-data table. This separates authentication from domain data and simplifies both RLS and the system's mental model.
2. **Sign-up and onboarding stay separate**: sign-up creates the account first, onboarding happens afterward. The app must not let users into the main flow if onboarding is still incomplete.
3. **Automatic profile creation on sign-up**: every new row in `auth.users` must automatically create a row in `public.profiles`, even if some fields start empty. This avoids inconsistent `user without profile` states.
4. **Minimal onboarding in v1**: onboarding only requires `username` and `team_id`. Avatar, bio, and the rest of the social profile stay out of the first cut.
5. **Teams as a real catalog**: `teams` will be a normal configurable table, not a rigid enum or a hardcoded app list.
6. **Three launch teams**: the MVP ships with 3 teams, but the database must be ready to support `N` teams later.
7. **Strong team identity from day one**: `teams` should support name, description, motto, colors, and logo from v1 because player attachment depends more on belonging than on a simple color.
8. **Team choice should carry real weight**: in v1, users cannot change teams after onboarding. A future change may exist, but only as an exceptional and expensive operation.
9. **Future team changes belong to the economy, not ranking**: if team switching is enabled later, the cost must come from the wallet and be recorded as a transaction, not paid with competitive points.
10. **Resource separation**: keep three layers separate from day one: recorded activity, competitive points, and wallet credits. One resource should not do everything.
11. **Wallet based on validated effort**: wallet credits come from validated effort, not from territorial impact.
12. **Competitive points based on territorial impact**: competitive points must depend mainly on strategic map value: unique zones traversed, zone state, and capture/defense events, with only a small base from distance.
13. **Normalized territorial scoring**: `zones` will not store rigid columns such as `points_red`, `points_blue`, or `points_yellow`. Team-level competitive state will live in a separate `zone_scores` table with one row per zone/team combination.
14. **Hybrid zone materialization for beta**: zones will not be created manually from the UI and the full territory will not be preloaded. The strategy is hybrid: limited pre-seed for the Community of Madrid and on-demand creation for the rest of the enabled beta territory.
15. **Territorial writes controlled by backend**: the client app must not write directly to `zones` or `zone_scores`. Zone creation and updates happen through controlled backend logic (Edge Function, RPC, or equivalent).
16. **Initial H3 resolution**: MVP beta will use H3 `resolution 9` as the compromise between urban granularity, map readability, and operational cost.
17. **Persisted geometry in `zones`**: besides `h3_index`, each zone should persist geometry useful for queries and map rendering instead of deriving it on the fly every time. A centroid may also be stored if it simplifies future rendering or querying.
18. **Simplified route in `activities`**: each activity stores a simplified route useful for the map, inspection, and territorial traceability. Raw unlimited GPS telemetry will not be the primary persisted source in v1.
19. **Compact anti-cheat summary inside `activities`**: v1 will not introduce a separate forensic table yet. Each activity should store enough validation summary to debug rejections and audit the basics without overdesigning the MVP.
20. **Activities private by default**: in beta, activities are only visible to their owner. Sharing with friends, followers, or external users is pushed to a later explicit social layer.
21. **Globally unique, case-insensitive `username`**: the player's public identity should be globally reserved without case ambiguity. That prepares the app for search, mentions, and a future social layer.
22. **Fixed `username` in v1**: users choose their `username` during onboarding and cannot change it in beta. Renaming remains a future decision outside the first cut.
23. **Email only in `auth.users`**: email must not be duplicated in `public.profiles`. Supabase Auth remains the canonical source, reducing drift, exposure of sensitive data, and RLS complexity.
24. **Wallet with ledger + materialized balance**: `transactions` is the source of accounting truth, while `profiles.wallet_balance` is a materialized balance for fast reads. Credit/debit operations must keep both consistent through controlled backend logic.
25. **Never allow negative balance in v1**: the wallet cannot go negative during beta. Any debit must validate sufficient balance before execution.
26. **Base transaction types in v1**: the ledger starts with `activity_reward` and `manual_adjustment` as real operating types. `team_change_fee` and `marketplace_spend` are reserved for future evolution.
27. **Whole-number credits only**: both `transactions.amount` and `profiles.wallet_balance` should use integers. The in-game economy operates in whole units to avoid rounding issues and simplify accounting.
28. **`amount` is always positive**: the ledger should not encode direction with a negative sign in `transactions.amount`. Movement direction is defined by `type`.
29. **Initial wallet formula**: beta starts with a deliberately simple reward based mostly on valid distance: `wallet_credits = round(valid_distance_km * 10)`. Invalid activities earn `0`.
30. **Economic expansion planned, not implemented in v1**: wallet rewards may later add streak bonuses, challenges, or temporary events, but not signals that mix economy with territorial competition.
31. **Territorial ownership updated immediately**: `owner_team_id` should be recalculated immediately after each valid activity inside the same backend logic that updates `zone_scores`. No periodic batch reconciliation in beta.
32. **Conservative ownership tie-breaker**: if there is a tie for highest score and the current owner is among the tied teams, the zone stays with that owner. If there is no current owner among the tied teams, the zone becomes neutral. Ownership only changes when one team leads alone.
33. **Visible ties in the UI**: even if `owner_team_id` stays the same under the conservative rule, the UI should indicate which teams are tied and why so users understand the zone is contested.
34. **No extra capture threshold in beta**: in v1, leading alone is enough to capture a zone. No additional gap over the second team is required.
35. **`zone_scores` initialized for every active team**: when a new zone is created, rows with `score = 0` should be created for all active teams. With 3 teams the cost is negligible and it simplifies queries, tie visualization, and neutral states.

## What Each Table Solves

### `teams`

Catalog of playable factions. Even though launch uses 3 teams, the table must not hardcode that as a technical rule.

Recommended minimum fields for v1:

- `id`
- `slug`
- `name`
- `motto`
- `description`
- `primary_color`
- `accent_color`
- `logo_url`
- `is_active`

### `profiles`

The player's product profile inside VICI. It lives in `public` because it represents product data, not the technical identity managed by Supabase Auth.

Recommended minimum fields for v1:

- `id` (same as `auth.users.id`)
- `username` nullable until onboarding is complete
- `team_id` nullable until onboarding is complete
- `wallet_balance` with default `0`
- `onboarding_completed_at` nullable
- `created_at`
- `updated_at`

`username` must have a global, case-insensitive uniqueness constraint to avoid collisions such as `CarlosRun` vs `carlosrun`.

Email lives only in `auth.users` and is not part of the profile's public identity.

### `activities`

Record of validated running sessions. This is where the fact that an activity happened lives. Economic reward and competitive impact derive from it, but they are not the same thing.

Expected minimum fields:

- `id`
- `user_id`
- `started_at`
- `ended_at`
- `distance_km`
- `duration_seconds`
- `avg_speed`
- `route` or equivalent simplified geometry
- `validation_status`
- `validation_reason` nullable
- `max_speed_kmh`
- `gps_point_count`
- `detected_anomalies` simple or nullable
- `wallet_credits_earned`
- `competitive_points_earned`
- `created_at`

An activity should store enough spatial context to explain why a run affected specific zones, without bloating storage with unlimited raw traces in the MVP.

More forensic detail or raw anti-cheat telemetry stays outside the first cut and can move to a separate table later if the game shows real abuse or manual-review needs.

### `zones`

Representation of map hexagons. It stores zone identity, summarized competitive state, and the minimum data needed to operate the territorial system.

Expected minimum fields:

- `h3_index`
- `owner_team_id` nullable
- `boundary` or equivalent hexagon geometry
- `centroid` optional if it helps rendering or queries
- `created_at`
- `updated_at`

`zones` stores identity and summarized state, not the detailed team-by-team scoring.

### `zone_scores`

Normalized table for per-zone, per-team scoring. It supports 3 teams today and `N` teams tomorrow without reshaping `zones`.

Expected minimum fields:

- `zone_h3_index`
- `team_id`
- `score`
- `updated_at`

The `zone_h3_index + team_id` combination must be unique.

When a new zone is created, `zone_scores` must be initialized with one row per active team and `score = 0`.

### `transactions`

Wallet ledger. Every inflow or outflow of credits must be recorded here, even if the aggregate balance is also reflected in `profiles.wallet_balance`.

Expected minimum fields:

- `id`
- `user_id`
- `amount`
- `type`
- `ref_activity_id` nullable
- `description`
- `created_at`

`transactions` acts as the auditable wallet ledger, while `profiles.wallet_balance` acts as a materialized snapshot for fast client reads.

In v1, no debit operation may leave `wallet_balance` below `0`.

`amount` is stored as an integer. No decimals are used for internal game credits in the MVP.

`amount` is always positive. Which types credit or debit is resolved by business rules based on `type`.

Agreed initial types:

- `activity_reward`
- `manual_adjustment`
- `team_change_fee` reserved for the future
- `marketplace_spend` reserved for the future

## Security and RLS in Product Terms

RLS (Row Level Security) means the database decides which rows a specific user can read or modify. In Supabase, this is not optional if the client app talks directly to the database.

Recommended initial policy for VICI v1:

- `teams`: open read, or at least readable by any authenticated user; no client-side writes
- `zones`: open read; no direct client writes
- `profiles`: each user may read and update only their own profile
- `activities`: each user may create and read only their own activities
- `transactions`: each user may read only their own transactions; creation must be tightly restricted and handled by backend logic

Future activity sharing should not be implemented by relaxing this baseline rule in the MVP. It should arrive as an extra social layer with explicit visibility rules and user relationships.

## Economy vs Competition

Three concepts must stay separate from now on:

- **Wallet credits**: spendable resource tied to validated effort
- **Competitive points**: non-spendable resource tied to strategic map impact
- **Zone points**: points that affect the state and ownership of a specific zone

This separation prevents currency spending from affecting competitive power and prevents ranking from becoming a simple reflection of wallet value.

The wallet follows a simple accounting pattern: the ledger lives in `transactions`, and the balance shown to the user is materialized in `profiles.wallet_balance`.

### Guiding Rule for v1

- `wallet_credits`: based on validated effort
- `competitive_points`: based mainly on unique zones traversed + zone state + capture/defense events, with only a small base from distance

### Initial Wallet Formula

For beta, the economic rule should be deliberately simple and explainable:

```text
wallet_credits = round(valid_distance_km * 10)
```

Associated rules:

- Valid activity earns credits based on validated distance
- Invalid activity earns `0` credits
- No complex multipliers yet for speed, heart rate, elevation, or territorial control

Examples:

- valid `3 km` -> `30` credits
- valid `5 km` -> `50` credits
- valid `10 km` -> `100` credits
- invalid activity -> `0` credits

### Accepted Future Expansions for the Wallet

These additions are consistent with the product economy and can be added later without breaking the base model:

- streak or consistency bonuses
- weekly challenge or event bonuses
- temporary multipliers controlled by campaigns or activations
- secondary effort signals such as validated moving time, if they prove meaningful

### Signals That Should Not Enter the Wallet Yet

These stay out of the wallet because they mix economy with competition, introduce noise, or encourage abuse:

- speed as a strong multiplier
- calories
- heart rate
- elevation as a primary factor
- unique zones, captures, or territorial ownership
- map ownership or competitive ranking

## Initial Privacy and Future Social Evolution

- Activities are private by default in beta
- Only the activity owner can read the full route and validation detail
- A future social layer may enable shared activities, follower/friend feeds, or partial visibility, but that is outside the first schema cut and outside initial RLS

## Beta Zone-Creation Strategy

Manual UI for `creating zones` is explicitly rejected. Zone materialization is a backend responsibility whenever a map query or finished activity detects hexagons that do not yet exist in the database.

### Agreed Strategy

- Initial pre-seed limited to the Community of Madrid
- On-demand creation for the rest of the territory enabled in beta
- `zones` and `zone_scores` writes only from backend
- H3 `resolution 9` as the base zone granularity

### Recommended Conceptual Flow

1. The app queries the map or sends a finished activity.
2. The backend calculates the involved `h3_index` values.
3. If a zone does not exist, it is created through `upsert`.
4. If the zone is new, `zone_scores` rows are initialized for all active teams with `0`.
5. `zone_scores` are read or updated as needed.
6. In the same operation, `owner_team_id` is recalculated to immediately reflect the resulting competitive state.

This reduces initial volume, avoids preloading unused territory, and preserves a living-world feeling in beta through localized pre-seeding.

Beta prioritizes immediate map feedback over deferred batch reconciliation.

### Initial Territorial Tie-Break Rule

- If one team leads alone, the zone belongs to that team
- If there is a tie for first place and the current owner is among the tied teams, the current owner keeps the zone
- If there is a tie for first place and no current owner is among the tied teams, the zone becomes neutral

This rule avoids erratic ownership changes and gives defense some weight without adding overly sophisticated logic.

There is no extra capture threshold in beta: leading alone is enough to become the owner.

### UI Requirement for Ties

The UI must be able to represent when a zone is tied or contested, even if `owner_team_id` does not change. The backend model should therefore leave enough traceability to answer questions such as:

- which teams are tied for first
- what each team's current score is
- why the zone keeps an owner or becomes neutral

This does not force the final visual design today, but it does mean ownership cannot be treated as the only relevant state of a zone.

## Technical Principles Guiding Implementation

1. **Do not mix auth and domain**: Supabase manages identity; VICI manages the player profile.
2. **Do not hardcode teams**: the catalog lives in the database.
3. **Do not allow economy/ranking drift**: keep resources separate.
4. **Do not delegate security to the UI**: real protection lives in RLS.
5. **Do not overcomplicate the MVP with premature flexibility**: the schema is ready to grow, but the first cut only solves what is necessary.
6. **Consistent public identity**: `username` must be stable, unique, and case-unambiguous.
7. **Sensitive data stays out of the public domain**: private auth data such as email must not be duplicated in product tables without a strong operational reason.
8. **Ledger first, fast reads second**: when a number is critical for audit and frequently displayed in UI, prioritize the ledger as truth and a materialized balance as controlled optimization.
9. **Defensive economy in the MVP**: simple and safe accounting rules are better than unnecessary flexibility in beta.
10. **Reserve future evolution without overdesigning**: reserved types or fields are fine when their cost is low and they avoid future redesign, but without implementing full functionality now.
11. **Simple accounting units for the MVP**: internal economic units are integers to reduce ambiguity and rounding errors.
12. **Explicit accounting semantics**: the movement type defines whether the ledger adds or subtracts; the `amount` field does not mix value and direction.
13. **Explainable economy**: the wallet formula must be easy to understand, communicate, and audit before it becomes more sophisticated.
14. **Immediate territorial feedback**: in beta, the map must reflect the effect of a valid activity without relying on deferred batch jobs.
15. **Simple territorial rules first**: while the game is validating its core loop, capture should follow rules that are easy to explain before adding thresholds and extra tuning layers.

### Territorial Scoring Formula

```text
competitive_points = (unique_zones * 10) + (captures * 25) + (defenses * 15) + round(distance_km * 1)
```

- Traversing a neutral zone counts as a unique zone, not a capture
- Conquering an enemy zone counts as a `capture`
- Traversing your own zone while staying the owner counts as a `defense`

### `zones` Geometry

- `boundary`: `geometry(Polygon, 4326)` PostGIS
- `centroid`: `geometry(Point, 4326)` PostGIS, persisted rather than derived on the fly to avoid recalculating it on every map render

### `activities.route` Format

- `geometry(LineString, 4326)` PostGIS
- Enables native spatial intersection queries (`ST_Intersects`) to calculate which H3 zones each activity crossed

### Minimum Anti-Cheat for v1

Initial anomaly catalog for `detected_anomalies`:

| Anomaly                   | Criteria                                                                           |
| ------------------------- | ---------------------------------------------------------------------------------- |
| `speed_exceeded`          | `avg_speed` or `max_speed` > `25 km/h`                                             |
| `insufficient_gps_points` | fewer than 1 point every 15 seconds for the declared duration                      |
| `teleport_detected`       | impossible instant jump between two consecutive points (`> 50 km/h` instant speed) |
| `accelerometer_validated` | **reserved for second iteration** - requires HealthKit/Google Fit integration      |

The accelerometer/step-detection check is reserved in the schema now but will not be implemented in beta. Native HealthKit/Google Fit integration is planned for weeks 14-16 of the roadmap and is the natural time to activate it.

### `activities` Write Flow

- Activity is created in the database at the **start of the running session** with status `pending`
- The client sends **periodic updates roughly every 1 km** during the session, updating `distance_km`, `duration_seconds`, `route`, and `gps_point_count` in the same row
- When the session ends, the backend processes the activity and transitions it to `valid` or `rejected`
- `validation_status` values in v1:

| State      | Meaning                                         |
| ---------- | ----------------------------------------------- |
| `pending`  | created or in progress, waiting to be processed |
| `valid`    | validated, rewards applied                      |
| `rejected` | validation failed, no rewards                   |

- A separate `activity_checkpoints` table stays outside the first cut. It can be revisited in the second iteration if anti-cheat needs or session recovery justify it.

### Exact RLS by Table

| Table          | SELECT                                 | INSERT        | UPDATE                                                | DELETE       |
| -------------- | -------------------------------------- | ------------- | ----------------------------------------------------- | ------------ |
| `teams`        | any authenticated user                 | backend only  | backend only                                          | backend only |
| `profiles`     | own row only (`id = auth.uid()`)       | trigger only  | own row only                                          | never        |
| `activities`   | own rows only (`user_id = auth.uid()`) | own rows only | own row only (checkpoint fields only while `pending`) | never        |
| `zones`        | any authenticated user                 | backend only  | backend only                                          | never        |
| `zone_scores`  | any authenticated user                 | backend only  | backend only                                          | never        |
| `transactions` | own rows only (`user_id = auth.uid()`) | backend only  | never                                                 | never        |

Important details:

- `activities.validation_status` is never writable from the client. Only backend logic (Edge Function or RPC) can transition `pending` -> `valid` / `rejected`. The `finish activity` button in the client calls an Edge Function; it does not perform a direct `UPDATE`.
- `transactions` inserts are backend-exclusive. The client never credits itself directly.

### Summarized Zone State for the Client

- Backend exposes a `zones_with_status` view that joins `zones` + `zone_scores` and adds derived fields: `is_contested`, `top_score`, `tied_team_ids`
- The client makes a single query against this view to render the map, with no territorial logic in the client
- If the tie rule changes later, only the view changes, not the app

### Real Wallet Operations in v1

- `activity_reward`: the only automatic credit mechanism. Executed by the Edge Function that processes each valid activity.
- `manual_adjustment`: ad hoc support operation that only admins can perform directly in the Supabase Dashboard during beta. No admin panel and no extra Edge Function in v1.
- The client never mutates balance directly.

### Extensions decision

- `h3-pg` is not available on the Supabase free tier — it requires a Pro plan upgrade.
- **Decision**: use `h3-js` (JavaScript library) in the app layer and Edge Functions for all H3 index calculations. The DB stores `h3_index` as `text`, which is sufficient for all MVP queries.
- PostGIS (`postgis` extension) was enabled via migration. It was installed into the `extensions` schema, so geometry columns use the `extensions.geometry(...)` qualified type.
- `h3-pg` is noted as a future Pro upgrade when query-level hex operations become a performance requirement.

### Migrations written and applied

Eight migrations were written in `packages/supabase/supabase/migrations/` and applied to the remote database with `supabase db push`:

| File                              | What it creates                                                                                                                                                                             |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `20260513000001_extensions.sql`   | `postgis` extension; sets `search_path` to include `extensions`; documents h3 as a future Pro upgrade                                                                                       |
| `20260513000002_teams.sql`        | `teams` table + RLS (authenticated read, no client writes)                                                                                                                                  |
| `20260513000003_profiles.sql`     | `profiles` table; `set_updated_at` trigger; `handle_new_user` trigger that auto-creates a profile row on `auth.users` insert; RLS (own row only for SELECT and UPDATE)                      |
| `20260513000004_zones.sql`        | `zones` table with `extensions.geometry` columns for boundary and centroid; GiST spatial indexes; RLS (authenticated read, no client writes)                                                |
| `20260513000005_zone_scores.sql`  | `zone_scores` table with composite PK `(zone_h3_index, team_id)`; non-negative score constraint; RLS (authenticated read, no client writes)                                                 |
| `20260513000006_activities.sql`   | `activities` table with `activity_validation_status` enum (`pending`, `valid`, `invalid`); GiST index on route; RLS (own rows for SELECT and INSERT, no client UPDATE of validation fields) |
| `20260513000007_transactions.sql` | `transactions` table with `transaction_type` enum; positive amount constraint; RLS (own rows for SELECT only, no client inserts)                                                            |
| `20260513000008_views.sql`        | `zones_with_scores` view — joins `zones` + `zone_scores`, aggregates team scores as JSON per zone                                                                                           |

**RLS note**: `activities.validation_status` and all `transactions` inserts are backend-only. The client never credits itself or transitions activity state directly.
